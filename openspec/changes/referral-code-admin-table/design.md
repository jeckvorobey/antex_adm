## Context

Админка должна показывать реферальные данные как часть общей таблицы пользователей, но без перегрузки интерфейса. Источник правды о доступности массовой генерации должен оставаться на backend, а не вычисляться только на клиенте.

## Goals / Non-Goals

**Goals:**
- Отобразить referral_code, процент начислений и balance в таблице пользователей.
- Показывать кнопку генерации только при наличии пользователей с `null` кодом.
- Сохранить текущие сценарии работы админки.

**Non-Goals:**
- Менять backend-логику генерации кодов.
- Переписывать всю админку или менять другие таблицы.
- Затрагивать не связанные с рефералкой экраны.

## Decisions

1. **Кнопка видна по данным backend-контракта.**
   - Это предотвращает рассинхрон между UI и данными.
   - Альтернатива: считать на фронте по строкам таблицы. Отклонено, потому что это ломается при пагинации и фильтрах.

2. **Новые поля добавляются в существующую таблицу как обычные колонки.**
   - Это проще для поддержки и не требует отдельного экрана.
   - Альтернатива: отдельный referral view. Отклонено как избыточное.

3. **Дизайн колонок должен учитывать длинные значения и responsive-сценарии.**
   - Без этого таблица может ломаться на узких экранах.
   - Альтернатива: скрывать часть данных. Отклонено, потому что эти данные нужны оператору.

## Risks / Trade-offs

- [Таблица может стать шире обычного] → использовать аккуратные width/overflow правила и проверить responsive.
- [Кнопка может быть недоступна при неполной пагинации] → опираться на backend-флаг, а не только на текущую страницу.
- [Старые тесты могут ожидать прежний набор колонок] → обновить тестовые ожидания вместе с UI.

## Migration Plan

1. Получить backend-контракт на список пользователей и доступность массовой генерации.
2. Обновить таблицу пользователей и кнопку в admin UI.
3. Прогнать visual/regression проверки.
4. Передать результаты в devops для следующего этапа delivery.
